# Кодстайл-гайд проекта devtools

Проект следует правилам `Google Java Style Guide` с адаптацией.
Автоматическая проверка: `./gradlew checkstyleMain`

## 1. Именование методов: camelCase

До:    `public void add_student(Student s) { }`

После: `public void addStudent(Student student) { }`

Почему: `Java Convention` требует `camelCase` для методов.

Источник: https://google.github.io/styleguide/javaguide.html#s5.3-camel-case

## 2. Пробелы после if/for/while

До:    `if(condition) {`

После: `if (condition) {`

Почему: улучшает читаемость, отделяет ключевое слово от выражения.

Источник: `Oracle Code Conventions — Whitespace`

## 3. Длина строки: максимум 120 символов

До:    `public List getStudentsFromSpecificCityWithVeryLongName...`

После: `public List getStudentsByCity(String city) {`

Почему: длинные строки затрудняют чтение в редакторе и при code review.

Источник: https://google.github.io/styleguide/javaguide.html#s4.4-column-limit

## 4. Порядок импортов

До:    `import java.util.List; import java.util.ArrayList; import java.io.File;`

После: `import java.io.File; import java.util.ArrayList; import java.util.List;`

Почему: алфавитный порядок упрощает поиск импортов.

Источник: `IntelliJ IDEA → Code → Optimize Imports`

## 5. Фигурные скобки для if

До:    `if (condition) doSomething();`

После: `if (condition) { doSomething(); }`

Почему: скобки обязательны даже для однострочных блоков.

Источник: https://google.github.io/styleguide/javaguide.html#s4.1.1-braces-always-used

### Промпт 1: План решения

**Задача:** Спроектируй класс студентов, чтобы можно было их добавлять без дублирования и различать, даже если у них одинаковыми данные.  
**Контекст:** Java 25. У меня есть record Student с полями email:String и name:String. Проект на Gradle.
**Ограничения:** Не используй регулярные выражения. Используй только встроенную библиотеку Java.
**Ожидаемый результат:** План из 3-5 шагов, объяснение "почему именно так", НЕ пиши код — я реализую сам
**Критерии успеха:** Это должен быть один класс в котором можно: добавить студента, удалить студента, вывести всех студентов.

**AI-1 ([Qwen3.6 Plus]):**

План: 4 шага

Шаг 1. Добавить уникальный идентификатор
В record Student добавить поле id (тип UUID или long). Генерировать его при создании. Это решает проблему различения студентов с одинаковыми email + name.

Почему UUID, а не автоинкремент: UUID потокобезопасен без синхронизации, не требует счётчика. Но если проект однопоточный — AtomicLong тоже подойдёт.

Шаг 2. Хранить студентов в Map<UUID, Student>. Ключ — id, значение — сам объект. Это даёт:
- O(1) на поиск/удаление
- Гарантия уникальности по ключу
- Быстрый get по ID

Почему не Set: Set не позволит добавить дубликат по equals(), но если equals завязан на ID — он всегда уникален, и Set не защитит от дублей по email+name. Map даёт контроль: ты сам решаешь, что считать дубликатом.
  
Шаг 3. Проверка дубликатов при добавлении
Перед вставкой проверить, нет ли уже студента с таким email. Для этого — дополнительный Map<String, Student> (email → Student) или проход по значениям основного Map.

Почему отдельный Map по email: O(1) вместо O(n) при проходе. Два индекса — классический паттерн.

Шаг 4. Удаление — по ID
Метод remove(UUID id) — удаляет из обоих Map'ов. Если нужно удаление по email — тоже возможно через второй индекс.

Итоговая структура класса:
StudentManager:
- \- Map<UUID, Student> byId
- \- Map<String, Student> byEmail
- \+ add(Student) → boolean / throws
- \+ remove(UUID) → boolean
- \+ getAll() → Collection<Student>
- \+ findById(UUID) → Optional<Student>
- \+ findByEmail(String) → Optional<Student>

Почему именно так:

|Требование|Решение|
|-|-|
|Без дубликатов|byEmail проверяет уникальность email|
|Различать одинаковых|id (UUID) — уникальный даже при совпадении всех полей|
|Добавить/удалить/вывести|Map даёт все три операции за O(1)|
|Без regex|Проверка email через String.isBlank() или isEmpty()|

**AI-2 ([DeepSeek]):**

План проектирования класса (без кода)

Шаг 1. Определите внутреннее хранилище на основе списка (ArrayList<Student>)

Почему:
Стандартный Set использует equals()/hashCode() record-класса Student, из-за чего объекты с одинаковыми email и name станут неразличимы для коллекции.
Список же позволяет хранить ссылки на объекты и управлять ими вручную.
Выбираем ArrayList – он сохраняет порядок добавления и даёт простой доступ по индексу.

Шаг 2. Реализуйте метод добавления с проверкой по идентичности (identity)

Почему:
Чтобы избежать дублирования одного и того же экземпляра (дважды передана одна ссылка), но разрешить добавление разных объектов с абсолютно одинаковыми полями.
Проверка должна идти по ==, а не по equals(). Итерация по списку и сравнение student == existing гарантирует это.

Шаг 3. Реализуйте метод удаления также по идентичности

Почему:
Если в списке есть несколько объектов с одинаковыми полями, удалить нужно именно тот, на который указывает переданная ссылка.
Используйте поиск по == (например, через removeIf или цикл с индексом), чтобы случайно не удалить другой «похожий» объект.

Шаг 4. Реализуйте метод вывода всех студентов

Почему:
Простой перебор элементов списка и вызов System.out.println (или подобного) для каждого Student.
Поскольку Student – record, его toString() уже отображает email и name. При одинаковых данных на экране будут одинаковые строки – это нормально, так как физически это разные объекты.

Шаг 5. (Опционально) Добавьте метод для получения текущего количества студентов

Почему:
Для удобства тестирования и отладки. Можно просто вернуть list.size(). Этот шаг не обязателен, но полезен, чтобы убедиться, что дублирование (по ссылке) действительно блокируется, а «одинаковые» студенты успешно добавляются.

Итоговое объяснение подхода

- Различение одинаковых данных обеспечивается за счёт использования проверки по ссылке (==), а не содержимому.
- Отсутствие дублирования трактуется как запрет на повторное добавление того же самого экземпляра, а не на добавление эквивалентного по полям.
- Такой дизайн полностью ложится на встроенные средства Java (без сторонних библиотек и регулярных выражений) и соответствует требованию «один класс» с тремя основными операциями: добавление, удаление, вывод.

**Сравнение:**
- AI-1 предложил уникализацию по UUID, что нагляднее и проще, чем сравнение объектов по хеш коду. 
- AI-1 предложил использовать Map, а AI-2 предложил использовать ArrayList. Первый вариант лучше в сочетании с UUID, а также оптимален из-за O(1).
- AI-1 предложил удалять по UUID, что проще и надежнее, чем удалять конкретный объект по хеш-коду.
- Рекомендации AI-1 точнее и лучше решают поставленную задачу.

```java
public String calculateTotalProgress(Mentee[] mentees) {
    if (mentees == null || mentees.length == 0) {
      throw new IllegalArgumentException("Массив менторов не должен быть пустой");
    }

    int totalCompleted = 0;
    int totalTotal = 0;
    int index = 0;

    while (index < mentees.length) {
      totalCompleted += mentees[index].completedLessons();
      totalTotal += mentees[index].totalLessons();
      index++;
    }

    int totalRemaining = totalTotal - totalCompleted;

    return "Суммарно: пройдено " + totalCompleted + " из " + totalTotal
        + " уроков, осталось " + totalRemaining + " уроков";
  }
```

**AI-1 ([Qwen3.6 Plus]):**

Review

Логические ошибки

1. Сообщение об ошибке не соответствует параметру — "Массив менторов не должен быть пустой", но параметр mentees (менти/студенты, не менторы).
2. Нет проверки элементов массива на null — если mentees[i] == null, будет NullPointerException на строке с completedLessons().
3. Возможное переполнение int — если суммарное количество уроков превысит Integer.MAX_VALUE (~2 млрд), произойдёт overflow. Для больших массивов стоит использовать long.
   
Code Conventions
1. while вместо enhanced for-loop — в Java принято использовать for (Mentee mentee : mentees). Ручной индекс — это C-стиль, не идиоматично для Java.
2. Имя переменной totalTotal — плохое имя, читается как тавтология. Лучше totalLessons или lessonsCount.
3. String-конкатенация — для одной строки компилятор сам оптимизирует в StringBuilder, но String.format() или текстовые блоки (Java 15+) были бы читаемее.

Производительность
- Критических проблем нет. Для одного вызова разница между while и for-each пренебрежима.
- Если массив большой (>100K элементов) и вызов частый — имеет смысл кэшировать результат или использовать long для сумм.

Рекомендации по улучшению

```java
/**
 * Рассчитывает суммарный прогресс всех учеников.
 *
 * @param mentees массив учеников (не должен быть null, не должен содержать null)
 * @return строка с суммарным прогрессом
 * @throws IllegalArgumentException если массив null, пуст или содержит null элементы
 */
public String calculateTotalProgress(Mentee[] mentees) {
    if (mentees == null || mentees.length == 0) {
        throw new IllegalArgumentException("Массив учеников не должен быть пустым");
    }

    int totalCompleted = 0;
    int totalLessons = 0;

    for (Mentee mentee : mentees) {
        if (mentee == null) {
            throw new IllegalArgumentException("Элемент массива учеников не может быть null");
        }
        totalCompleted += mentee.completedLessons();
        totalLessons += mentee.totalLessons();
    }

    int totalRemaining = Math.max(0, totalLessons - totalCompleted);
    return String.format("Суммарно: пройдено %d из %d уроков, осталось %d уроков",
                         totalCompleted, totalLessons, totalRemaining);
}
```
