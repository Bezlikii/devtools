# Кодстайл-гайд проекта devtools

Проект следует правилам `Google Java Style Guide` с адаптацией.
Автоматическая проверка: `./gradlew checkstyleMain`

## 1. Именование методов: camelCase

До:    `public void add_student(Student s) { }`

После: `public void addStudent(Student student) { }`

Почему: `Java Convention` требует `camelCase` для методов.

Источник: https://google.github.io/styleguide/javaguide.html#s5.3-camel-case

## 2. Пробелы после if/for/while

До:    `if(condition) {`

После: `if (condition) {`

Почему: улучшает читаемость, отделяет ключевое слово от выражения.

Источник: `Oracle Code Conventions — Whitespace`

## 3. Длина строки: максимум 120 символов

До:    `public List getStudentsFromSpecificCityWithVeryLongName...`

После: `public List getStudentsByCity(String city) {`

Почему: длинные строки затрудняют чтение в редакторе и при code review.

Источник: https://google.github.io/styleguide/javaguide.html#s4.4-column-limit

## 4. Порядок импортов

До:    `import java.util.List; import java.util.ArrayList; import java.io.File;`

После: `import java.io.File; import java.util.ArrayList; import java.util.List;`

Почему: алфавитный порядок упрощает поиск импортов.

Источник: `IntelliJ IDEA → Code → Optimize Imports`

## 5. Фигурные скобки для if

До:    `if (condition) doSomething();`

После: `if (condition) { doSomething(); }`

Почему: скобки обязательны даже для однострочных блоков.

Источник: https://google.github.io/styleguide/javaguide.html#s4.1.1-braces-always-used

### Промпт 1: План решения

**Задача:** Спроектируй класс студентов, чтобы можно было их добавлять без дублирования и различать, даже если у них одинаковыми данные.  
**Контекст:** Java 25. У меня есть record Student с полями email:String и name:String. Проект на Gradle.
**Ограничения:** Не используй регулярные выражения. Используй только встроенную библиотеку Java.
**Ожидаемый результат:** План из 3-5 шагов, объяснение "почему именно так", НЕ пиши код — я реализую сам
**Критерии успеха:** Это должен быть один класс в котором можно: добавить студента, удалить студента, вывести всех студентов.

**AI-1 ([Qwen3.6 Plus]):**

План: 4 шага

Шаг 1. Добавить уникальный идентификатор
В record Student добавить поле id (тип UUID или long). Генерировать его при создании. Это решает проблему различения студентов с одинаковыми email + name.

Почему UUID, а не автоинкремент: UUID потокобезопасен без синхронизации, не требует счётчика. Но если проект однопоточный — AtomicLong тоже подойдёт.

Шаг 2. Хранить студентов в Map<UUID, Student>. Ключ — id, значение — сам объект. Это даёт:
- O(1) на поиск/удаление
- Гарантия уникальности по ключу
- Быстрый get по ID

Почему не Set: Set не позволит добавить дубликат по equals(), но если equals завязан на ID — он всегда уникален, и Set не защитит от дублей по email+name. Map даёт контроль: ты сам решаешь, что считать дубликатом.
  
Шаг 3. Проверка дубликатов при добавлении
Перед вставкой проверить, нет ли уже студента с таким email. Для этого — дополнительный Map<String, Student> (email → Student) или проход по значениям основного Map.

Почему отдельный Map по email: O(1) вместо O(n) при проходе. Два индекса — классический паттерн.

Шаг 4. Удаление — по ID
Метод remove(UUID id) — удаляет из обоих Map'ов. Если нужно удаление по email — тоже возможно через второй индекс.

Итоговая структура класса:
StudentManager:
- \- Map<UUID, Student> byId
- \- Map<String, Student> byEmail
- \+ add(Student) → boolean / throws
- \+ remove(UUID) → boolean
- \+ getAll() → Collection<Student>
- \+ findById(UUID) → Optional<Student>
- \+ findByEmail(String) → Optional<Student>

Почему именно так:

|Требование|Решение|
|-|-|
|Без дубликатов|byEmail проверяет уникальность email|
|Различать одинаковых|id (UUID) — уникальный даже при совпадении всех полей|
|Добавить/удалить/вывести|Map даёт все три операции за O(1)|
|Без regex|Проверка email через String.isBlank() или isEmpty()|

**AI-2 ([DeepSeek]):**

План проектирования класса (без кода)

Шаг 1. Определите внутреннее хранилище на основе списка (ArrayList<Student>)

Почему:
Стандартный Set использует equals()/hashCode() record-класса Student, из-за чего объекты с одинаковыми email и name станут неразличимы для коллекции.
Список же позволяет хранить ссылки на объекты и управлять ими вручную.
Выбираем ArrayList – он сохраняет порядок добавления и даёт простой доступ по индексу.

Шаг 2. Реализуйте метод добавления с проверкой по идентичности (identity)

Почему:
Чтобы избежать дублирования одного и того же экземпляра (дважды передана одна ссылка), но разрешить добавление разных объектов с абсолютно одинаковыми полями.
Проверка должна идти по ==, а не по equals(). Итерация по списку и сравнение student == existing гарантирует это.

Шаг 3. Реализуйте метод удаления также по идентичности

Почему:
Если в списке есть несколько объектов с одинаковыми полями, удалить нужно именно тот, на который указывает переданная ссылка.
Используйте поиск по == (например, через removeIf или цикл с индексом), чтобы случайно не удалить другой «похожий» объект.

Шаг 4. Реализуйте метод вывода всех студентов

Почему:
Простой перебор элементов списка и вызов System.out.println (или подобного) для каждого Student.
Поскольку Student – record, его toString() уже отображает email и name. При одинаковых данных на экране будут одинаковые строки – это нормально, так как физически это разные объекты.

Шаг 5. (Опционально) Добавьте метод для получения текущего количества студентов

Почему:
Для удобства тестирования и отладки. Можно просто вернуть list.size(). Этот шаг не обязателен, но полезен, чтобы убедиться, что дублирование (по ссылке) действительно блокируется, а «одинаковые» студенты успешно добавляются.

Итоговое объяснение подхода

- Различение одинаковых данных обеспечивается за счёт использования проверки по ссылке (==), а не содержимому.
- Отсутствие дублирования трактуется как запрет на повторное добавление того же самого экземпляра, а не на добавление эквивалентного по полям.
- Такой дизайн полностью ложится на встроенные средства Java (без сторонних библиотек и регулярных выражений) и соответствует требованию «один класс» с тремя основными операциями: добавление, удаление, вывод.

**Сравнение:**
- AI-1 предложил уникализацию по UUID, что нагляднее и проще, чем сравнение объектов по хеш коду. 
- AI-1 предложил использовать Map, а AI-2 предложил использовать ArrayList. Первый вариант лучше в сочетании с UUID, а также оптимален из-за O(1).
- AI-1 предложил удалять по UUID, что проще и надежнее, чем удалять конкретный объект по хеш-коду.
- Рекомендации AI-1 точнее и лучше решают поставленную задачу.

### Промпт 2: Проверка кода

**Код для проверки:**

```java
public String calculateTotalProgress(Mentee[] mentees) {
    if (mentees == null || mentees.length == 0) {
      throw new IllegalArgumentException("Массив менторов не должен быть пустой");
    }

    int totalCompleted = 0;
    int totalTotal = 0;
    int index = 0;

    while (index < mentees.length) {
      totalCompleted += mentees[index].completedLessons();
      totalTotal += mentees[index].totalLessons();
      index++;
    }

    int totalRemaining = totalTotal - totalCompleted;

    return "Суммарно: пройдено " + totalCompleted + " из " + totalTotal
        + " уроков, осталось " + totalRemaining + " уроков";
  }
```

**AI-1 ([Qwen3.6 Plus]):**

Review

Логические ошибки

1. Сообщение об ошибке не соответствует параметру — "Массив менторов не должен быть пустой", но параметр mentees (менти/студенты, не менторы).
2. Нет проверки элементов массива на null — если mentees[i] == null, будет NullPointerException на строке с completedLessons().
3. Возможное переполнение int — если суммарное количество уроков превысит Integer.MAX_VALUE (~2 млрд), произойдёт overflow. Для больших массивов стоит использовать long.
   
Code Conventions
1. while вместо enhanced for-loop — в Java принято использовать for (Mentee mentee : mentees). Ручной индекс — это C-стиль, не идиоматично для Java.
2. Имя переменной totalTotal — плохое имя, читается как тавтология. Лучше totalLessons или lessonsCount.
3. String-конкатенация — для одной строки компилятор сам оптимизирует в StringBuilder, но String.format() или текстовые блоки (Java 15+) были бы читаемее.

Производительность
- Критических проблем нет. Для одного вызова разница между while и for-each пренебрежима.
- Если массив большой (>100K элементов) и вызов частый — имеет смысл кэшировать результат или использовать long для сумм.

```java
/**
 * Рассчитывает суммарный прогресс всех учеников.
 *
 * @param mentees массив учеников (не должен быть null, не должен содержать null)
 * @return строка с суммарным прогрессом
 * @throws IllegalArgumentException если массив null, пуст или содержит null элементы
 */
public String calculateTotalProgress(Mentee[] mentees) {
    if (mentees == null || mentees.length == 0) {
        throw new IllegalArgumentException("Массив учеников не должен быть пустым");
    }

    int totalCompleted = 0;
    int totalLessons = 0;

    for (Mentee mentee : mentees) {
        if (mentee == null) {
            throw new IllegalArgumentException("Элемент массива учеников не может быть null");
        }
        totalCompleted += mentee.completedLessons();
        totalLessons += mentee.totalLessons();
    }

    int totalRemaining = Math.max(0, totalLessons - totalCompleted);
    return String.format("Суммарно: пройдено %d из %d уроков, осталось %d уроков",
                         totalCompleted, totalLessons, totalRemaining);
}
```

### Отладка — Debug

**Определение:** Процесс поиска и устранения дефектов и проблем в программном обеспечении. В контексте IDE отладка означает запуск программы в специальном режиме, который позволяет разработчику приостановить выполнение, проверить значения переменных и пошагово пройти код строка за строкой.

**Контекст использования:** Debug используется для поиска логических ошибок в коде. В IntelliJ IDEA запускаем Debug (`Shift+F9`), ставим Breakpoint на проблемной строке, проверяем значения переменных через Evaluate Expression.

**Пример:** Метод `calculateProgress` возвращает неверный результат. Ставим Breakpoint на строке с расчётом, запускаем Debug, проверяем значения `completedLessons` и `totalLessons` через Evaluate Expression (`Alt+F8`), находим ошибку в логике цикла.

**Источник:** https://www.jetbrains.com/help/idea/debugging-your-first-java-application.html (статус: 200, дата проверки: 12.05.2026)

### Точка останова — Breakpoint

**Определение:** Маркер, установленный на конкретной строке исходного кода, который приостанавливает выполнение программы во время отладки, позволяя разработчику проверить текущее состояние переменных, стек вызовов и поток выполнения в этой точке.

**Контекст использования:** Breakpoint используется для остановки выполнения программы на конкретной строке, чтобы проверить значения переменных, пошагово пройти код и понять логику работы программы. В IntelliJ IDEA Breakpoint обозначается красным кружком на полях редактора.

**Пример:** В методе `calculateProgress` ставим Breakpoint на строке `while (index < mentees.length)`, запускаем Debug, программа останавливается на Breakpoint, в панели Variables видим текущие значения `index`, `totalCompleted`, `totalTotal`.

**Источник:** https://www.jetbrains.com/help/idea/debugging-your-first-java-application.html (статус: 200, дата проверки: 12.05.2026)

### JaCoCo

**Определение:** Бесплатная библиотека анализа покрытия кода тестами для Java. JaCoCo измеряет, какая часть кода выполняется тестами, и формирует отчёты по покрытию строк, веток и методов, помогая командам гарантировать достаточное качество тестирования.

**Контекст использования:** JaCoCo анализирует, какой процент кода покрыт тестами. В проекте devtools настроен минимальный порог покрытия 80% (исключая `*Main.class`). JaCoCo запускается через Gradle и интегрируется с CI.

**Пример:** Выполняем `./gradlew test` — JaCoCo создаёт отчёт в `build/reports/jacoco/test/`. Открываем `index.html` и видим: покрытие строк — 85%, покрытие веток — 78%. Если покрытие ниже 80%, команда `./gradlew check` завершится ошибкой.

**Источник:** https://www.jacoco.org/jacoco/trunk/doc/index.html (статус: 200, дата проверки: 12.05.2026)

### GitHub Actions

**Определение:** Платформа CI/CD, интегрированная в GitHub, позволяющая автоматизировать процессы сборки, тестирования и развёртывания непосредственно из репозитория. Workflows определяются в YAML-файлах в каталоге `.github/workflows/` и запускаются по событиям, таким как push, pull request или по расписанию.

**Контекст использования:** GitHub Actions автоматически запускает проверки при каждом коммите или Pull Request: Checkstyle, тесты, сборку. Конфигурация CI для проекта devtools находится в `.github/workflows/ci.yml`.

**Пример:** Отправляем Push в ветку `feature/DVT-12` — GitHub Actions автоматически запускает workflow: `./gradlew checkstyleMain`, `./gradlew test`, `./gradlew build`. Если все проверки пройдены, в PR отображается зелёная галочка.

**Источник:** https://docs.github.com/en/actions/about-github-actions (статус: 200, дата проверки: 12.05.2026)

## Процессы и практики

### Коллегиальное ревью — Peer Review

**Определение:** Практика проверки, при которой один или несколько коллег исследуют представленный код или документ для выявления дефектов, предложения улучшений и обмена знаниями. Peer Review опирается на человеческое суждение и обсуждение, в отличие от автоматизированных проверок, которые применяют стилистические или структурные правила.

**Контекст использования:** Peer Review — это этап, когда коллеги-разработчики вручную проверяют код перед слиянием. Цель: найти логические ошибки, предложить улучшения архитектуры, поделиться знаниями. Peer Review дополняет Automated Review (Checkstyle, тесты).

**Пример:** После создания Pull Request `feature/DVT-12` → `master`, наставник или коллега просматривает код: проверяет логику методов, предлагает вынести дублирующийся код в отдельный метод, отмечает, что стоит добавить тест на граничный случай.

**Источник:** https://www.agilealliance.org/agile101/agile-glossary/ (статус: 200, дата проверки: 12.05.2026)

---

### Автоматизированное ревью — Automated Review

**Определение:** Процесс использования программных инструментов для автоматической проверки кода на стилистические нарушения, потенциальные ошибки, покрытие тестами и другие метрики качества без участия человека. Automated Review запускается при каждом коммите или PR и предоставляет быструю, объективную обратную связь.

**Контекст использования:** Automated Review выполняется инструментами, настроенными в CI: Checkstyle проверяет стиль кода, JaCoCo — покрытие тестами, JUnit запускает тесты. Это быстрый и объективный первый фильтр перед Peer Review.

**Пример:** В проекте devtools при каждом Push в Pull Request GitHub Actions запускает `./gradlew checkstyleMain test` — это Automated Review. Если Checkstyle находит нарушение или тест падает, PR не может быть слит. После Automated Review проходит Peer Review.

**Источник:** https://checkstyle.org/ (статус: 200, дата проверки: 12.05.2026), https://docs.github.com/en/actions (статус: 200, дата проверки: 12.05.2026)

---

### Непрерывная интеграция — CI

**Определение:** Практика разработки, при которой разработчики регулярно интегрируют код в общий репозиторий. Каждая интеграция проверяется автоматической сборкой и тестами, что позволяет командам быстро обнаруживать ошибки и предотвращать проблемы интеграции.

**Контекст использования:** CI автоматически проверяет каждый коммит: компилирует код, запускает тесты и статический анализ. В проекте devtools CI реализован через GitHub Actions — при каждом Push или PR запускаются проверки.

**Пример:** Разработчик выполняет Push в ветку `feature/DVT-12` — GitHub Actions автоматически запускает `./gradlew checkstyleMain test build`. Если тесты падают или Checkstyle находит нарушения — CI красный, PR нельзя слить. Все проверки зелёные — можно проводить Peer Review.

**Источник:** https://www.atlassian.com/continuous-delivery/continuous-integration (статус: 200, дата проверки: 12.05.2026). Примечание: источник Martin Fowler (https://martinfowler.com/bliki/ContinuousIntegration.html) недоступен (404, дата проверки: 12.05.2026), использована альтернатива.

---

### Непрерывная доставка/развёртывание — CD

**Определение:** Расширение непрерывной интеграции, обеспечивающее готовность программного обеспечения к выпуску в любой момент. Continuous Delivery означает, что каждое изменение, прошедшее автоматические тесты, может быть развёрнуто с ручным подтверждением. Continuous Deployment идёт дальше и автоматически развёртывает каждое успешное изменение на production.

**Контекст использования:** CD — следующий уровень после CI. Continuous Delivery: код готов к развёртыванию в любой момент, но требует ручного подтверждения. Continuous Deployment: код автоматически развёртывается на production без ручного вмешательства. Проект devtools пока использует только CI.

**Пример:** В проекте devtools настроен уровень CI (автоматические тесты и Checkstyle при каждом коммите). Для перехода к Continuous Delivery нужно добавить: автоматическую сборку артефакта, публикацию в репозиторий, развёртывание на staging-сервере. Для Continuous Deployment — автодеплой на production после зелёного CI.

**Источник:** https://www.atlassian.com/continuous-delivery (статус: 200, дата проверки: 12.05.2026)

---

### Runbook

**Определение:** Документированный набор процедур и инструкций для эксплуатации и устранения неисправностей в системе. Runbook содержит пошаговые инструкции для типичных операционных задач, реагирования на инциденты и процедур восстановления, позволяя дежурным инженерам быстро решать проблемы.

**Контекст использования:** Runbook помогает быстро реагировать на типичные проблемы: упавшие тесты, ошибки сборки, проблемы с зависимостями. В проекте devtools Runbook описывает сценарии ручной проверки и пути решения частых ошибок.

**Пример:** Разработчик запускает `./gradlew build` и получает ошибку. Открывает Runbook, находит секцию «Что делать, если сборка падает», следует инструкциям: проверяет версию JDK, очищает кэш (`./gradlew clean`), проверяет `JAVA_HOME`. Проблема решена за 2 минуты вместо 30.

**Источник:** https://sre.google/sre-book/table-of-contents/ (статус: 200, дата проверки: 12.05.2026)

---

### Спринт — Sprint

**Определение:** Ограниченный по времени период (обычно 1–4 недели), в течение которого должен быть завершён и подготовлен к ревью определённый набор работ. В Scrum Sprint — это основная итерация, включающая планирование, выполнение, обзор и ретроспективу.

**Контекст использования:** Sprint разбивает проект на короткие итерации с конкретной целью. В учебном проекте «Java с нуля до оффера» Sprint 3 — текущий спринт для модуля Dev Tools. В конце Sprint проводится ретроспектива и планирование следующего.

**Пример:** В проекте devtools Sprint 3 включает задания DVT-10, DVT-11, DVT-12. Каждый Sprint начинается с Planning (определение задач) и заканчивается Review (демо результатов) и Retrospective (что улучшить в следующем Sprint).

**Источник:** https://scrumguides.org/scrum-guide.html (статус: 200, дата проверки: 12.05.2026)


# Вопросы по сложным терминам

## Вопрос 1: В чём разница между Continuous Integration (CI) и Continuous Delivery/Deployment (CD)?

**Задача:** Разобраться, чем отличается непрерывная интеграция от непрерывной доставки/развёртывания — где заканчивается CI и начинается CD, и почему их часто объединяют как «CI/CD».

**Контекст:** В проекте devtools настроен GitHub Actions, который запускает тесты и Checkstyle при каждом коммите — это CI. В уроке DVT-12 термин CI/CD указан как один термин, но CI и CD — разные понятия с разными целями.

**Ограничения:** Прочитал определения в Atlassian Docs и Google SRE Book, но остаются неясными границы: автоматическое тестирование — это уже CI, а автодеплой на сервер — это уже CD? Где граница между Continuous Delivery и Continuous Deployment?

**Ожидаемый результат:** Чётко понимать: CI — автоматическая сборка и тестирование при каждом коммите; CD (Delivery) — автоматическая подготовка релиза к развёртыванию; CD (Deployment) — автоматическое развёртывание на production без ручного подтверждения. Уметь объяснить, к какому уровню относится наш GitHub Actions в проекте.

**Критерии успеха:** Могу назвать 3 ключевых различия между CI и CD, могу определить, на каком уровне CI/CD находится конкретный проект (наш devtools — на уровне CI), и объяснить коллеге, что нужно добавить в пайплайн, чтобы перейти от CI к CD.


## Вопрос 2: В чём разница между Agile и Scrum?

**Задача:** Понять, чем Agile отличается от Scrum — являются ли они одним и тем же, или Scrum это часть Agile, и какие ещё подходы используют разработчики в Agile помимо Scrum.

**Контекст:** В обсуждениях командной работы часто встречаются оба термина, иногда как синонимы. В уроке DVT-12 упоминается Sprint как термин из процессов, но неясно, относится ли он к Agile или к Scrum конкретно.

**Ограничения:** Ознакомился с Agile Manifesto (agilemanifesto.org) и Scrum Guide (scrumguides.org), но осталось непонимание: Agile — это философия с 4 ценностями и 12 принципами, а Scrum — конкретный фреймворк с ролями (Scrum Master, Product Owner, Developer), событиями (Sprint, Daily, Retro) и артефактами (Backlog). Как они соотносятся на практике?

**Ожидаемый результат:** Понимать, что Agile — это набор ценностей и принципов (философия), а Scrum — один из фреймворков, реализующих Agile-принципы. Уметь назвать минимум 2 других Agile-фреймворка (Kanban, XP) и объяснить как их используют разработчики, с чем можно столкнуться в компаниях.

**Критерии успеха:** Могу объяснить коллеге разницу между Agile и Scrum, а также понимать какие подходы используются в коммерческой разработке.
